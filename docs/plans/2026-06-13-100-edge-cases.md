# Аудит: 100 багов и edge cases (только документирование)

## Overview

Системный аудит проекта с целью найти и задокументировать 100 багов / edge cases. Результат — единый структурированный реестр в `docs/audit/2026-06-13-100-edge-cases.md`. Правок кода НЕТ — только находки (severity, file:line, repro, предлагаемый фикс). Формат повторяет существующий `docs/audit/2026-05-23-audit-findings.md`.

## Context

- Output file: `docs/audit/2026-06-13-100-edge-cases.md` (новый)
- Образец формата: `docs/audit/2026-05-23-audit-findings.md`
- Зоны охвата (по CLAUDE.md): XP/Levels/Streaks, SRS, Daily Plan, Curriculum/Lessons/Grading, Books/Reading, Achievements/Race/Challenge, Notifications/Feedback, Admin, API/Security/Middleware, Infrastructure, Frontend/templates/CSS, контент модулей (JSON)
- Методика: чтение кода + grep по edge-сигнатурам (None/[]/0/boolean, naive vs aware datetime, race/idempotency, off-by-one, тайзоны, деление на ноль, флаги precedence), запуск `pytest` / `pytest -m smoke` для фиксации красных тестов, сверка с уже известными failing tests (memory: ~52 red на master)
- Severity: P0 (data-loss/security/сломанный сценарий), P1 (UX-баг, тех-долг с риском), P2 (полировка/dead code)

## Development Approach

- **Testing approach**: N/A — это аудит, кода не меняем, тестов не пишем. «Verify» в каждой задаче = подтвердить, что каждая находка ссылается на реальный `file:line` и воспроизводима по коду (grep/read), без ложных срабатываний.
- Каждая задача — sweep одного доменного кластера, дописывает батч находок в единый реестр с уникальными id (E-001…E-100+).
- Распределение квоты ~ по 12-15 находок на доменную задачу, чтобы суммарно перевалить за 100.
- Дубли и «не баг» отсекаются в финальной консолидации.
