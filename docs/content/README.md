# docs/content

Documentation for the content layer of the immersion rollout
(`docs/plans/2026-05-11-post-immersion-content-data-plan.md`). The
folder holds three kinds of documents:

1. Acceptance contracts. Numbers that gate production rollout — the
   importers, audits, and gap reports are written against these.
   - `immersion-content-targets.md` — slot saturation, CEFR seed
     minimums, audio quality requirements, vocabulary enrichment
     targets, SRS source coverage, acceptance criteria.
2. Schema references. Per-lesson-type payload contracts used by
   `LessonContentValidator` and the importers.
   - `lesson-content-schemas.md` (Task 9).
3. Index README files. This file.

Everything procedural — backups, rollouts, on-call steps — lives in
`docs/runbooks/`. Plans live in `docs/plans/`. Reviews and ADRs live
in their own directories under `docs/`.

## What belongs in `docs/content`

- Lesson type schemas, including required + optional fields, grader
  expectations, and example payloads.
- Content target matrices that define acceptance numbers.
- Content authoring conventions (style guides, CEFR length bands,
  audio quality requirements).
- Cross-references between content files, importers, audits, and
  reports.

## What does NOT belong here

- Lesson content itself. Lessons live in `content/immersion/`.
- Vocabulary data. CSVs live in `content/vocabulary/`.
- Implementation plans. Plans live in `docs/plans/`.
- Step-by-step operational procedures. Those live in
  `docs/runbooks/`.

## Report generation conventions

All reports live in `reports/`. They are produced by scripts under
`scripts/` and follow a uniform shape.

### Naming

| Pattern                                  | Source script                                  |
| ---------------------------------------- | ---------------------------------------------- |
| `reports/immersion_data_audit.md`        | `scripts/audit_immersion_data.py`              |
| `reports/migration_readiness.md`         | (manual; produced from `pytest tests/migrations/`) |
| `reports/seed_coverage.md`               | (manual; from `app/achievements/seed.py` inspection) |
| `reports/module_insertion_map.md`        | Narrative companion to `content/immersion/module_insertion_map.csv` |
| `reports/immersion_gap_report.md`        | `scripts/report_immersion_gaps.py`             |
| `reports/<lesson_type>_import.md`        | `scripts/import_immersion_lessons.py` (per-run report)  |
| `reports/cefr_seed_import.md`            | `scripts/import_immersion_lessons.py` aggregated  |
| `reports/existing_listening_payloads.md` | `scripts/audit_existing_listening_payloads.py` |
| `reports/listening_duration_transcript_audit.md` | `scripts/audit_existing_listening_payloads.py` (audio check mode) |
| `reports/vocabulary_enrichment.md`       | `scripts/report_vocabulary_enrichment.py`      |
| `reports/admin_content_quality.md`       | (manual; produced from admin dashboard snapshot) |
| `reports/staging_immersion_rollout.md`   | (manual; record of Task 49)                    |
| `reports/production_immersion_rollout.md`| (manual; record of Task 50)                    |

### Required structure

Every generated report includes:

1. A title (`# <Subject>`).
2. A "generated" stanza:
   ```
   _Generated by `<script_path>` on YYYY-MM-DD UTC._
   ```
   Manual reports use `_Compiled on YYYY-MM-DD by <author>._`.
3. A short context paragraph linking back to the plan task that
   produced the report.
4. Sections with numeric findings first, narrative second. Tables
   beat prose for counts.
5. A "Next actions" or "Acceptance" section that maps findings back
   to the acceptance contract in
   `immersion-content-targets.md` where applicable.

### CLI contract for generators

All report-producing scripts follow the same flags:

- `--output PATH` — overrides the default `reports/<name>.md`.
- `--format markdown|json` — markdown by default.
- `--dry-run` — for importers, prints intended changes without
  writing.

### Idempotency

Reports must be regenerable. The same script invocation against the
same DB state must produce identical output (modulo timestamps).
Reviewers rely on this to compare runs across rollouts.

## Stable id conventions across content

- Lesson rows: `external_key` in lesson JSON, format
  `{lesson_type}:{level}:{module_number:02d}:{slug}` — see
  `content/immersion/README.md`.
- Vocabulary rows: `word_id` referencing `collection_words.id`.
- Module identity: `(level, module_number, source_filename)`
  triple, with the canonical source under `module_completed/fixed/`.
- DailyChallenge: `(challenge_date, lesson_id)` — no external id.

Stable ids are the contract between content authors and the database.
Once a lesson is shipped, its `external_key` is frozen.
