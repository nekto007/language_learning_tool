# v1 Scope and Explicit Deferrals

## Status: LOCKED (Phase 0, 2026-04-18)

---

## What Is In v1

v1 is the smallest set of features needed to test H1, H2, H3, and H5.

- **"Day secured" state**: boolean flag in the daily plan payload set when all required phases are complete. Visible banner on dashboard. Logged with timestamp.
- **1 next-best-step recommendation**: a single recommended continuation task shown after day_secured. Human-readable reason string. Returned from GET /api/daily-plan/next-step. Priority: unfinished lesson > SRS due > grammar weak > reading progress > vocab. Returns None when genuinely exhausted.
- **Basic route board (position + checkpoint)**: UserRouteProgress model tracking weighted steps and checkpoints. Route state exposed in daily plan API. Dashboard shows progress bar and checkpoint label.
- **Ghost rivals (adult only, opt-in)**: one ghost rival per user, deterministic position, labeled "Training Rival". No real-user comparison. Adults only, gated by birth year or explicit adult flag. Permanently dismissable.

## Deferred to v2+

The following features are not in v1. They must not influence v1 architecture decisions.

- **Endless queue engine**: generating an unbounded list of next steps. v1 returns at most 3 ordered options. Full queue generation requires anti-burnout pacing logic, batch generation, and quality scoring across the entire card pool — all of which depend on Phase 2 validation before they are worth building.
  *Rationale*: H4 (does endless queue outperform single next step?) cannot be tested until H1 is validated. Building the engine before that result is waste.

- **Batch generation**: pre-generating tomorrow's plan in background jobs. Requires a stable v1 data model and usage patterns before the batch cadence, invalidation logic, and content freshness rules can be defined correctly.
  *Rationale*: premature optimization; adds operational complexity before the content selection logic is stable.

- **Full rival matching**: matching users against real other users by skill segment, streak, or plan_completion count. Requires enough daily active users in similar segments to produce fair matches. Also requires significant trust and fairness testing before real-user data is shown.
  *Rationale*: ghost rivals test the motivational signal cheaply. Real-user matching adds social risk (shame, gaming) that is only worth the cost if ghost rivals validate the hypothesis.

- **Seasonal leagues**: grouping users into time-bounded competition seasons with promotion/relegation. Requires a real rival pool, a defined scoring system, and enough engagement history to balance league tiers.
  *Rationale*: depends on full rival matching (v2+) being proven before adding this additional layer of complexity.

- **Boss rounds**: special high-difficulty sessions unlocked at checkpoints or league milestones.
  *Rationale*: requires validated route and rival systems first. Do not design boss mechanics until the route model is stable.

- **Speed rounds**: timed sessions as a variation of review mode.
  *Rationale*: a UX experiment that can be run independently of this plan. Not needed to test any H1–H5 hypothesis.

- **Mixed-block mode**: interleaving vocabulary, grammar, and reading cards within a single session.
  *Rationale*: instructional validity work (Task 4) defined phase transitions carefully. Mixed-block changes those transitions in ways that have not been validated. Defer until the single-phase model is proven.

## Deferred to v3+

- **Real-time rivalry**: live progress updates showing a rival's route position changing in the current session. Requires WebSocket infrastructure and raises significant stress-inducing UX risk.
  *Rationale*: synchronous competition is not necessary for H3. Ghost rivals already test the motivational signal. Real-time adds infrastructure cost and harm risk that belongs in v3 after extensive testing.

- **Overtake events**: push notifications or in-session alerts when a rival passes the user. Raises anxiety risk that anti-shame rules specifically prohibit.
  *Rationale*: explicitly forbidden by anti-shame guardrails in Task 3 and Task 4. Revisit only if Phase 3 data shows no stress signal and adult users explicitly request it.

- **Child vs adult product divergence**: separate content queues, reward curves, and UX for minors. Requires dedicated child-safety review, parental consent mechanisms, and separate A/B testing.
  *Rationale*: v1 simply excludes children from rivals (gated by birth year). Full divergence is a separate product track that must not delay adult v1 shipping.

## Architecture Constraints: Nothing in v1 Must Block Deferrals

The following rules apply to all v1 implementation decisions:

1. The next-step API (GET /api/daily-plan/next-step) must return a list, not a scalar, so the response envelope supports extending to 3 or more items in v2 without a breaking change.
2. The UserRouteProgress model must use a tunable `CHECKPOINT_STEPS` constant, not a hardcoded value, so the checkpoint interval can be adjusted based on Phase 2 data.
3. Ghost rival position must be computed from a deterministic seed, not stored per-user, so that the real-rival model (v2+) can replace it without a migration of ghost rows.
4. The `day_secured` flag must be computed from the mission required-phase list, not from a hardcoded step count, so that the definition of "minimum" can evolve without touching the secured detection logic.
5. No v1 endpoint or model may assume a single next step. All data structures must be collection-friendly even when v1 returns at most 1 item.
